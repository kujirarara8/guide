Chapter 01

GitHubとは？

ひとことで言うと、インターネット上の「ファイル保管庫＋共同作業スペース」。
世界中のクリエイターやエンジニアが使う、業界標準のプラットフォームです。

  • HTMLファイル、画像、コードなどを保存・管理できるサービス
  • 「Git（変更履歴を管理する仕組み）」＋「Hub（みんなが集まる場所）」という意味
  • 基本機能はすべて無料で使える
Image

Googleドライブのクリエイター版のような存在。ただし「誰が・いつ・どこを変えたか」がすべて履歴として残るのが大きな特徴です。

Chapter 02

GitHubを使う意味

GitHubの中でも、特に便利なのがGitHub Pagesという機能。
これを使えば、HTMLファイルをそのままWebページとして公開できます。

01
無料でWebページが公開できる
HTMLをアップロードするだけで、自分専用のURLが発行され、誰でもアクセスできるページになります。サーバー契約もドメイン契約も不要です。
02
AIで作ったHTMLをそのまま公開できる
ClaudeやChatGPTで生成したHTMLや資料、シミュレーターなどを、数分でWebページ化してシェアできます。
03
変更履歴がすべて残る
「前のバージョンに戻したい」「いつ誰が変えたか確認したい」がすぐできます。ミスをしても元に戻せる安心感があります。
04
複数人で同じプロジェクトを管理できる
チームメンバーと同じファイルを共同編集でき、バージョンがバラバラにならずに一元管理できます。
05
安定していて長期運用に向いている
他の無料公開サービスに比べてアクセス制限が緩く、長く使い続けられる安定性があります。
Chapter 03

事前準備：アカウントを作る

初回のみ、5分ほどで完了します。

i

サインアップ

GitHubの公式サイトにアクセスし、右上の「Sign up」をクリック。以下を順番に入力していきます。

  • メールアドレス
  • パスワード（15文字以上、または8文字以上＋数字＋小文字）
  • ユーザー名（これが公開URLに使われるので重要）

認証パズルとメール認証コードを入力すれば完了です。

ii

ユーザー名の決め方

公開URLは必ず https://ユーザー名.github.io/... の形になります。つまり、ユーザー名がそのまま公開URLの一部になるので、じっくり考えて決めましょう。

  • 短く、英数字とハイフンのみ（日本語・記号はNG）
  • ビジネス名やブランド名に関連するものがおすすめ
  • 一度決めたら変更しにくいので、愛着が持てる名前に
Chapter 04

基本操作：ページを公開する

アカウントができたら、いよいよページを公開していきます。
実際の操作は、たった4ステップで完了します。

1

リポジトリ（箱）を作る

リポジトリとは、ファイルを入れるフォルダのこと。プロジェクトごとに1つ作ります。

操作手順

  • ログイン後、画面右上の「＋」マーク →「New repository」をクリック
  • Repository name（リポジトリ名）を入力：英数字とハイフンで命名
  • 公開設定：必ず「Public」を選択（GitHub Pagesは公開リポジトリ限定）
  • 「Add a README file」にチェック（推奨）
  • 下の「Create repository」ボタンをクリック
Tip

URLを第三者に知られたくない場合は、リポジトリ名に推測されにくい英数字を混ぜましょう（例：my-project-k3m9p2）。URLを知っている人だけがアクセスできる状態になります。

2

ファイルをアップロードする

操作手順

  • 作ったリポジトリ画面の「Add file」→「Upload files」をクリック
  • パソコンからHTMLファイルをドラッグ＆ドロップ
  • 画面下の「Commit changes」ボタンをクリック
Important

トップページとなるファイル名は、必ず index.html にしてください。これがルールです。画像やCSS、サブページは自由に追加できます。

フォルダ構成の例

my-project/
├── index.html ← トップページ（必須）
├── about.html ← 別ページ（任意）
├── style.css ← デザインファイル
└── images/
    └── hero.jpg ← 画像
Note

「Commit（コミット）」とは「この変更を記録する」という意味。変更内容をメモ欄に書いておくと、後から履歴を見るときに便利です（例：「初回アップロード」）。

3

GitHub Pagesを有効化する

ここが公開のスイッチです。必ず実施してください。

操作手順

  • リポジトリ画面の上部メニューから「Settings」をクリック
  • 左メニューの「Pages」をクリック
  • 「Build and deployment」セクションで、Source を「Deploy from a branch」に
  • Branch を「main」に、フォルダは「/ (root)」のまま
  • 「Save」ボタンをクリック

数分待つと、画面上部に公開URLが表示されます。

4

URLを確認する

公開URLの形式

https://ユーザー名.github.io/リポジトリ名/

ユーザー名の部分は自分のアカウント名、リポジトリ名の部分は手順1で作成した名前に置き換わります。

Patience

初回の公開反映までは、3〜10分程度かかることがあります。「404 Not Found」が出ても焦らず、しばらく待ってから再度アクセスしてみてください。

Chapter 05

応用：更新・追加・削除・履歴

日常運用でよく使う操作をまとめました。

A. ファイルを修正する

  • リポジトリ画面で編集したいファイル名をクリック
  • 右上の鉛筆アイコン（Edit this file）をクリック
  • ブラウザ上で直接編集し、「Commit changes」で保存
  • 1〜3分で公開ページにも反映されます

B. ファイルを追加する

「Add file」→「Upload files」または「Create new file」から。ドラッグ＆ドロップでまとめて追加もできます。

C. ファイルを削除する

削除したいファイルをクリック → 右上のゴミ箱アイコン →「Commit changes」で確定。

D. 過去のバージョンに戻す

ファイル画面の「History」をクリック → 戻したい時点のコミットを選択 → 内容を確認して復元。

Chapter 06

公開したページを共有する

公開されたURLは、LINEやDMでそのまま送るだけ。
相手に特別な準備や登録は一切必要ありません。

項目 内容
アカウント登録不要
GitHubログイン不要
スマホ閲覧可能
パスワードなし（Publicリポジトリの場合）
Chapter 07

外部公開・お客様に見せるときのポイント

GitHub Pagesで公開するページは、世界中の誰からもアクセスできる状態になります。
お客様や受講生さんに見てもらう前に、ここだけは必ず確認しておきましょう。

公開前の最終チェックリスト

  • 個人情報・内部情報（電話番号、住所、未公開の金額など）が載っていないか
  • 誤字脱字・リンク切れがないか、最初から最後まで目を通す
  • スマホ・PC両方で表示確認をする（お客様の多くはスマホで開きます）
  • 画像が重すぎて読み込みが遅くなっていないか
  • 使用している画像・フォントの著作権は大丈夫か（商用利用OKか）
Caution

PublicリポジトリはURLを知らなくても検索されればたどり着ける可能性があります。機密情報や限定公開の内容は載せないのが鉄則です。「知られても問題ないもの」だけを公開しましょう。

お客様に見てもらうときの工夫

01
URLを短く・覚えやすくする
リポジトリ名は公開URLの一部になります。お客様にシェアする前提なら、できるだけシンプルで短い名前にしましょう。長すぎると読み間違いや入力ミスの原因に。
02
QRコードを作って案内する
チラシ・名刺・店頭POPなどに載せる場合は、URLをQRコード化すると便利。無料のQRコード作成サービスで、数秒で作れます。
03
開いたときの第一印象を整える
ページを開いた瞬間に目に入る部分（ファーストビュー）は特に丁寧に。画像・タイトル・キャッチコピーで「これは何のページか」が3秒で伝わるように設計しましょう。
04
公開直後は自分で一度アクセスする
公開してすぐに、自分のスマホや別のブラウザから実際にURLを開いて動作確認。「表示されない」「画像が出ない」などはここで気づけます。
05
更新したら必ず再確認
お客様に送ったあとで修正した場合、キャッシュで古いページが表示されることも。更新後は一度スーパーリロードで最新状態を確認してから案内を。
Tip

お客様から「見られない」「URLが違う」と言われたときのために、自分でURLをコピーしてメモ帳などに貼る習慣をつけておくと安心です。入力ミスでURLを間違えて伝えるのを防げます。

Chapter 08

よくあるつまずきと解決法

ページが表示されない（404エラー）
  • トップページのファイル名が index.html になっているか確認
  • Pages設定のBranchが「main」になっているか確認
  • 5〜10分待ってから再度アクセス
画像が表示されない
  • ファイル名に日本語・スペースが入っていないか確認
  • HTML内のパス指定が正しいか確認（例：images/hero.jpg）
更新が反映されない
  • ブラウザのスーパーリロード（Windows：Ctrl+F5 / Mac：Cmd+Shift+R）
  • 1〜3分待ってから再確認
Chapter 09

困ったら、AIに聞くのもおすすめ

GitHubに限らず、今の時代はClaudeやChatGPTなどのAIが、とても頼もしい相談相手になります。
「わからない」を抱え込まず、気軽に聞いてみる習慣が、上達への一番の近道です。

AIに聞けること（具体例）

01
エラーが出たときの原因と解決策
「404エラーが出ました」「画像が表示されません」など、画面の状態をそのまま伝えれば、原因と対処法を提案してくれます。スクリーンショットを貼るとさらに精度アップ。
02
HTMLの編集・カスタマイズ
「この部分の色を変えたい」「文字を大きくしたい」「ボタンを追加したい」など、やりたいことを日本語で伝えれば、HTMLコードを書いてくれます。コピペで即反映。
03
用語の意味を優しく教えてもらう
「コミットって何？」「リポジトリとフォルダの違いは？」など、わからない言葉はその場で質問。専門用語も、自分のレベルに合わせて説明してくれます。
04
文章や見出しのブラッシュアップ
公開前のページ原稿を「もっと読みやすく」「お客様に響く表現に」など、一緒に磨いていけます。ターゲット層を伝えるとさらに精度の高い提案が返ってきます。
05
次にやるべきことの相談
「ページを公開しました。次に何をすればいいですか？」と聞けば、運用や改善のアイデアまで提案してくれます。一人で抱え込まず、伴走者として活用しましょう。

AIに質問するときのコツ

  • やりたいことをシンプルに伝える（例：「このHTMLのタイトルの色を赤にしたい」）
  • 今起きている状況を具体的に書く（例：「保存したのに更新されません」）
  • HTMLコードやエラーメッセージはそのままコピペして貼る
  • 一度で完璧な答えが出なくても、追加で質問を重ねて調整していく
Mindset

「こんな初歩的なこと聞いていいのかな」と思う必要は一切ありません。AIは疲れも機嫌も持たず、何度聞いても丁寧に答えてくれます。わからないまま放置するのが一番もったいないので、気軽に話しかけてみてください。

Finale

まとめ

最短4ステップで、あなたのページは公開できる

  1. リポジトリを作る（Public設定で）
  2. index.html を含むファイル一式をアップロード
  3. Settings → Pages → Branchを「main」に設定
  4. 数分待って、発行されたURLで確認

一度、自分の手で公開までやってみてください。
2回目からは、5分でスマートに公開できるようになります。
